# Техническое задание: Telegram-бот «Мистер Кек»

> ТЗ составлено на основе реверс-инжиниринга существующего кода (`index.js`, `constants.js`) для переписывания проекта с нуля.

## 1. Назначение и концепция

«Мистер Кек» — Telegram-бот для группового чата, реализующий игровую механику виртуальной валюты **«кек»** (аналог лайка/реакции «смешно»).

Суть: участники чата награждают друг друга «кеками» за смешные сообщения. Все кеки складываются в общую «Кеказну», по которой строится рейтинг — кто в чате самый смешной. Бот ведёт учёт баланса каждого участника и предоставляет дополнительные развлекательные механики (отзыв кека, «трипл-кек», кек-казино).

Стиль общения бота — нарочито грубый, ироничный, с жаргоном (целевая аудитория — закрытый дружеский чат).

## 3. Два Telegram-клиента (ключевая архитектурная особенность)

Бот использует **одновременно два подключения**:

1. **Bot-клиент** (Telegraf, токен `BOT_TOKEN`) — приём команд, текстовых сообщений, отправка ответов в чат.
2. **User-клиент** (GramJS / MTProto, `StringSession` + `API_ID` + `API_HASH`) — работает от имени **реального пользовательского аккаунта**, потому что Bot API не умеет:
   - искать медиа по истории канала (`messages.Search`);
   - читать историю сообщений между ID (`messages.GetHistory`);
   - получать список участников канала (`channels.GetParticipants`);
   - отправлять медиа из канала (`messages.SendMedia`);
   - (закомментировано) ставить реакции на сообщения.

> ⚠️ **Критическая проблема безопасности в текущем коде:** строка сессии (`sessionKey`) **захардкожена прямо в `index.js`**. Это полный доступ к аккаунту. В новой версии — обязательно вынести в секреты/переменные окружения и ротировать ключ.

## 5. Модель данных (`db.json`)

```jsonc
{
  "users": [
    {
      "id": 448341870,            // Telegram user id
      "name": "Дима",              // отображаемое имя
      "kekNumber": 100,            // баланс кеков (старт — 100)
      "lastKekGivenTo": {          // последний выданный кек (для отзыва), либо null
        "userId": 337052957,
        "messageId": 12345
      }
    }
  ],
  "messagesWithKek": [
    {
      // полный объект сообщения, которому поставили кек, плюс:
      "kekedUsers": [372958499]    // id пользователей, поставивших кек этому сообщению
    }
  ]
}
```

Инициализация (`initDB`): `users` заполняется из участников чата, каждому `kekNumber = 100`, `lastKekGivenTo = null`; `messagesWithKek = []`.

## 8. Игровые механики (обработка обычных сообщений)

На каждое входящее сообщение проверяются три типа триггеров (кек / некек / казино) и выполняется соответствующая обработка.

### 8.1. Поставить кек (`processKekMessage`)
1. Определить целевое сообщение:
   - если текущее сообщение — **reply**, цель = сообщение, на которое ответили;
   - иначе цель = «предыдущее осмысленное» сообщение в чате (поиск через историю, пропуская служебные кек/некек-сообщения и сообщения самого бота без ID автора).
2. Определить автора целевого сообщения. Если автор — бот, ID реального автора извлекается из текста по шаблону `(<число>)`; если не найден — кек перенаправляется на LUX с шуткой.
3. **Запрет самокека:** нельзя ставить кек собственному сообщению.
4. Если цель сама не является кек-сообщением — выдать кек (`giveKek`):
   - у дарителя `kekNumber -= 1`, у получателя `kekNumber += 1`;
   - у дарителя сохраняется `lastKekGivenTo = {userId, messageId}`;
   - сообщение регистрируется в `messagesWithKek`, в `kekedUsers` добавляется ID дарителя (без дублей);
   - публичное уведомление в чат.
5. **Трипл-кек:** если у сообщения накопилось **3** кека от разных пользователей — особое поздравление автору, сообщение удаляется из `messagesWithKek`.
6. После обработки — очистка устаревших записей (`clearOldMessagesWithKek`): из `messagesWithKek` удаляются сообщения старше **24 часов** (по полю `date`).

### 8.2. Забрать кек / некек (`revertKek`)
- Отзывает **последний** выданный пользователем кек (`lastKekGivenTo`).
- Возврат балансов: дарителю `+1`, получателю `-1`.
- Из `kekedUsers` соответствующего сообщения удаляется ID; если список опустел — запись удаляется.
- `lastKekGivenTo` обнуляется.
- Граничные случаи: «ты ещё никому не давал кек»; «у получателя 0 кеков — отжать нельзя» (с шуткой про «рабство»).

### 8.3. Кек-казино (`onRandomKek` / `/kekcasino`)
1. Требует инициализации (`/start`).
2. Взнос — **1 кек** (списывается сразу).
3. Через user-клиент ищется всё медиа (фото/видео) в основном чате (`messages.Search` с пагинацией по 100 до полной выборки).
4. Выбирается случайное медиа, отправляется в промежуточный канал с подписью игрока, затем копируется в исходный чат.
5. При ошибке — кек возвращается игроку, сообщение об ошибке.

## 9. Рейтинг (`collectUserStats`)
Список пользователей с балансами; для LUX — особый «почётный» титул, для остальных — насмешливый. Формат строки: `<титул> <имя> - <N> кеков` (HTML).

## 10. Жизненный цикл приложения
1. Параллельный старт user-клиента (`botUserClient.start()`) и bot-клиента (`bot.launch()`).
2. Загрузка `db.json`; если данных нет — `initDB`.
3. Логирование `Bot started`.
4. Graceful shutdown по `SIGINT` / `SIGTERM`.

## 11. Известные баги текущей версии (исправить при переписывании)
1. **Сессия захардкожена** в исходниках (критично, см. §3).
2. `giveKek`: проверка остатка использует `fromUserId.kekNumber` вместо `fromUser.kekNumber` — баланс по факту не проверяется (можно уйти в минус).
3. `collectUserStats`: компаратор `sort` возвращает boolean вместо числа — сортировка рейтинга некорректна.
4. `resetStats`: `users.map(...)` создаёт новый массив, но результат не присваивается — **сброс не работает**.
5. ID участников и админ зашиты в код — не масштабируется на другие чаты.
6. Промежуточный канал казино (`-1001493761518`) и `animeKonfaId` зашиты в код.
7. Нет обработки конкурентного доступа к `db.json` (гонки при одновременных операциях).
8. `onBotStart` обращается к `localDB.data.users` без проверки на инициализацию.
